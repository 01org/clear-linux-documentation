Enable DPDK support on OpenvSwitch

Enabling DPDK support on the OpenvSwitch project can yield significant network performance improvements. To illustrate one such example, we’ll cover the simple use case shown on figure 1, where one virtual machine sends 1,000,000 HTTP requests to another virtual machine. The connection between both VMs will be through a virtual bridge created by different tools:

  • Linux bridges.
  • OpenvSwitch bridges.
  • Custom-built of OpenvSwitch-DPDK bridges.

Figure 1: Basic virtual network environment.

Requirements

  1. The recommended release for this example is greater or equal to 13360, it is possible to update the system with this command.

    # swupd update

  2. Install the following bundles: network-basic, kvm-host.

    # swupd bundle-add network-basic
# swupd bundle-add kvm-host

  3. Download a clear linux image and OVMF.fd file, this image will be used as the guest VMs in the Clear Linux downloads.

Using Linux Bridges

  1. Create a script named qemu-ifup for a Linux bridge in a virtual machine.

    #!/bin/bash

set -x

switch=br0

if [ -n "$1" ];then
    # tunctl -u `whoami` -t $1 (use ip tuntap instead!)
    ip tuntap add $1 mode tap user `whoami`
    ip link set $1 up
    sleep 0.5s
    brctl addif $switch $1
    exit 0;
else
    echo "Error: no interface specified"
    exit 1
fi

  2. Enable execute permission on the script, preserving its group permissions with respect to other files.

    # chmod a+x qemu-ifup

  3. Create a bridge using the brctl tool; you can verify whether the bridge was successfully created by using ip tool.

    # brctl addbr br0

    Note

    At this point, it is possible to add a NIC with brctl addif br0 <network interface>, example:

    # brctl addif br0 enp3s0f0

  4. Set up the Linux bridge.

    # ip link set dev br0 up

  5. Run guest virtual machine A using the next configuration as reference, where the $IMAGE var is the Clear Linux image name.

    $ qemu-system-x86_64 \
      -enable-kvm -m 1024 \
      -bios OVMF.fd \
      -smp cpus=2,cores=1 -cpu host \
      -vga none -nographic \
      -drive file="$IMAGE",if=virtio,aio=threads \
      -net nic,macaddr=00:11:22:33:44:55,model=virtio -net tap,script=qemu-ifup \
      -debugcon file:debug.log -global isa-debugcon.iobase=0x402

  6. Run guest virtual machine B using the configuration from the previous step; take care to update the MAC address.

  7. Follow the instructions from the Setting IP Address section.

  8. Alternatively, clean the previous environment, turn off the virtual machines, and delete the bridge.

    # ip link set dev br0 down
# brctl delbr br0

Using OpenvSwitch

  1. Start the OpenvSwitch service.

    # systemctl start openvswitch.service

  2. Create a bridge using the OpenvSwitch tool; you can verify whether or not the bridge was successfully created by running ip tool.

    # ovs-vsctl add-br br0
# ip a

  3. Create an UP script named ovs-ifup which can bring up the tap devices.

    #!/bin/sh

switch="br0"
/usr/bin/ifconfig $1 0.0.0.0 up
ovs-vsctl add-port ${switch} $1

  4. Create a DOWN script named ovs-ifdown which can bring up the tap devices.

    #!/bin/sh

switch="br0"
/usr/bin/ifconfig $1 0.0.0.0 down
ovs-vsctl del-port ${switch} $1

  5. Enable execute permission on the scripts, preserving their group permissions with respect to other files.

    # chmod a+x ovs-ifdown
# chmod a+x ovs-ifup

  6. Run guest virtual machine A using the next configuration as reference, where $IMAGE var is the name of the Clear Linux* OS for Intel Architecture image. Notice the network configuration uses the up-down scripts.

    $ qemu-system-x86_64 \
      -enable-kvm -m 1024 \
      -bios OVMF.fd \
      -smp cpus=2,cores=1 -cpu host \
      -vga none -nographic \
      -drive file="$IMAGE",if=virtio,aio=threads \
      -net nic,model=virtio,macaddr=00:11:22:33:44:55 -net tap,script=ovs-ifup,downscript=ovs-ifdown \
      -debugcon file:debug.log -global isa-debugcon.iobase=0x402

  7. Run guest virtual machine B using the configuration from step 5, only it’s necessary to change the MAC address to something like 00:11:22:33:44:56

  8. Follow the instructions in the Setting IP address section.

  9. Alternatively, clean the previous environment, turn off the virtual machines, and delete the bridge.

    # ovs-vsctl del-br br0
# ovs-vsctl show

Using Linux OpenvSwitch-DPDK

  1. Enable VT-d technology in the BIOS.

  2. Enable VT-d in the host kernel command line, to enable VT-d in the host kernel command line, the clear-linux-native-current-kernel-version.conf file must be edited. Add iommu=pt intel_iommu=on to the end of the line. The file is found within the UEFI boot partition.

    # systemctl start boot.mount
# cd /boot/loader/entries/

  3. Unmount the UEFI partition and reboot the machine.

    # cd /
# systemctl stop boot.mount
# reboot

  4. Set number of hugepages.

    # echo 1024 > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages

  5. Allocate pages on NUMA machines.

    # echo 1024 > /sys/devices/system/node/node0/hugepages/hugepages-2048kB/nr_hugepages
# echo 1024 > /sys/devices/system/node/node1/hugepages/hugepages-2048kB/nr_hugepages

  6. Make memory available for the DPDK.

    # mkdir -p /mnt/huge
# mount -t hugetlbfs nodev /mnt/huge

  7. Download a clear linux image and OVMF.fd file, this image will be used as the guest VMs in the Clear Linux downloads.

  8. Start the OpenvSwitch service.

    # systemctl start openvswitch

  9. OpenvSwitch must be configured to enable the DPDK functionality like core mask, socket memory, and others. This example reproduces the environment shown in figure 1.0. The OpenvSwitch documentation provides additional information about DPDK configuration.

    # ovs-vsctl --no-wait init
# ovs-vsctl --no-wait set Open_vSwitch . other_config:dpdk-lcore-mask=0x2
# ovs-vsctl --no-wait set Open_vSwitch . other_config:dpdk-socket-mem=2048
# ovs-vsctl --no-wait set Open_vSwitch . other_config:dpdk-init=true

  10. Restart the OpenvSwitch service in order to update the new DPDK configuration.

    # systemctl restart openvswitch

  11. Create a virtual bridge using openvswitch.

    # ovs-vsctl add-br br0 -- set bridge br0 datapath_type=netdev

  12. Add the vhost-dpdk ports to the bridge.

    # ovs-vsctl add-port br0 vhost-user1 -- set Interface vhost-user1 type=dpdkvhostuser
# ovs-vsctl add-port br0 vhost-user2 -- set Interface vhost-user2 type=dpdkvhostuser

  13. Run guest virtual machine A using the next configuration as reference, where $IMAGE var is the name of the Clear Linux* OS for Intel Architecture image.

    $ qemu-system-x86_64 \
    -enable-kvm -m 1024 \
    -bios OVMF.fd \
    -smp 4 -cpu host \
    -vga none -nographic \
    -drive file="$IMAGE",if=virtio,aio=threads \
    -chardev socket,id=char1,path=/run/openvswitch/vhost-user1 \
    -netdev type=vhost-user,id=mynet1,chardev=char1,vhostforce \
    -device virtio-net-pci,mac=00:00:00:00:00:01,netdev=mynet1 \
    -object memory-backend-file,id=mem,size=1024M,mem-path=/dev/ hugepages,share=on \
    -numa node,memdev=mem -mem-prealloc \
    -debugcon file:debug.log -global isa-debugcon.iobase=0x402

  14. Run guest virtual machine B, use the configuration from the previous step; simply change the MAC address and the port socket. You can use 00:00:00:00:00:02 as a MAC address and vhost-user2 as a socket.

  15. Follow the instructions from the Setting IP address section.

Setting IP address

  1. Set an IP address to virtual machine for virtual machine A:

    # ip addr add dev enp0s2 10.0.0.5/24

    for virtual machine B:

    # ip addr add dev enp0s2 10.0.0.6/24

  2. Check if there is communication between both virtual machines using ping tool.

  3. Verify that Apache service is running:

    # systemctl status httpd.service
# systemctl start httpd.service

    Start httpd service only if it is inactive. Use apache benchmark to get information about the network performance between both virtual machines.

    # ab -n 1000000 -c 100 http://10.0.0.6/
