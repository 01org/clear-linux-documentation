OpenvSwitch and DPDK

Introduction

Enabling DPDK support on the OpenvSwitch project can yield significant network performance improvements. To illustrate one such example, we’ll cover a simple use case with Figure 1: Basic virtual network environment., where one virtual machine sends 1,000,000 HTTP requests to another virtual machine using * Linux bridges* * OpenvSwitch bridges * And custom-built OpenvSwitch-DPDK bridges as the network link.

Figure 1: Basic virtual network environment.

Requirements:

To follow along with this example, you will need:

  • At least one platform using Clear Linux* OS for IntelÂ® Architecture (recommended release >= 7160) as Host,
  • The kernel-native bundle enabled on that particular Host,
  • And the network-advanced, and os-clr-on-clr bundles installed.
# swupd bundle-add network-advanced os-clr-on-clr

You’ll also need two kvm images (recommended release >= 7160). These images will create the guest virtual machines A and B. These virtual machines must also have network-basic and lamp-basic bundles installed.

# swupd bundle-add network-basic lamp-basic

Using Linux Bridges

  1. Create a script named qemu-ifup for a linux bridge in a virtual machine.

    #!/bin/bash
      set -x
      switch=br0
       if [ -n "$1" ];then
          tunctl -u whoami -t $1
          ip link set $1 up
          sleep 0.5s
          brctl addif $switch $1
          exit 0
       else
          echo "Error: no interface specified"
          exit 1
       fi

  2. Enable execute permission on the script, preserving its group permissions with respect to other files.

    # chmod a+x qemu-ifup

  3. Create a bridge using the OpenvSwitch tool; you can verify whether the bridge was successfully created by using ip tool.

    # brctl addbr br0
Note: At this point, you have the option to add a NIC with brctl addif br0 <network interface>. Sample interface below should be replaced with your local NIC’s designation.
# brctl addif br0 enp3s0f0

When the above option is used, and the NIC is connected to DHCP server, so Steps 1 and 2 under the Setting IP address section.

  1. Set up the Linux bridge.

    # ip link set dev br0 up

  2. Run guest virtual machine A using the next configuration as reference, where the $IMAGE var is the Clear Linux image name.

    qemu-system-x86_64 \
  -enable-kvm -m 1024 \
  -bios OVMF.fd \
  -smp cpus=2,cores=1 -cpu host \
  -vga none -nographic \
  -drive file="$IMAGE",if=virtio,aio=threads \
  -net nic,macaddr=00:11:22:33:44:55,model=virtio -net tap,script=qemu-ifup \
  -debugcon file:debug.log -global isa-debugcon.iobase=0x402

  3. Run guest virtual machine B using the configuration from the previous step; take care to update the MAC address.

  4. Follow the instructions from the Setting IP Address section.

  5. And to clean the previous environment, turn off the virtual machines and delete the bridge.

    # ip link set dev br0 down
# brctl delbr br0

Using OpenvSwitch

  1. Start the OpenvSwitch service.

    # systemctl start openvswitch.service

  2. Create a bridge using the OpenvSwitch tool; you can verify whether or not the bridge was successfully created by running ip tool.

    # ovs-vsctl add-br br0
# ip a

  3. Create UP-DOWN scripts which can bring up the tap devices through the bridge we created in Step 2.

    The ovs-ifdown script should look something like:

    #!/bin/sh
 switch="br0"
 /usr/bin/ifconfig $1 0.0.0.0 down
 ovs-vsctl del-port ${switch} $1

    And the ovs-ifup script should look something like:

    #!/bin/sh
 switch="br0"
 /usr/bin/ifconfig $1 0.0.0.0 up
 ovs-vsctl add-port ${switch} $1

  4. Enable execute permission on the scripts, preserving their group permissions with respect to other files.

    # chmod a+x ovs-ifdown
# chmod a+x ovs-ifup

  5. Run guest virtual machine A using the next configuration as reference, where $IMAGE var is the name of the Clear Linux* OS for Intel Architecture image. Notice the network configuration uses the up-down scripts.

    qemu-system-x86_64 \
    -enable-kvm -m 1024 \
    -bios OVMF.fd \
    -smp cpus=2,cores=1 -cpu host \
    -vga none -nographic \
    -drive file="$IMAGE",if=virtio,aio=threads \
    -net nic,model=virtio,macaddr=00:11:22:33:44:55 -net tap,script=ovs-ifup,downscript=ovs-ifdown \
    -debugcon file:debug.log -global isa-debugcon.iobase=0x402

  6. Run guest virtual machine B using the configuration from step 5, only it’s necessary to change the MAC address to something like 00:11:22:33:44:56

  7. Follow the instructions in the Setting IP address section.

Using Linux OpenvSwitch-DPDK

  1. Create a new environment and add iommu=pt intel_iommu=on to the kernel command line.

    # systemctl start boot.mount
# cd /boot/loader/entries/

  2. Edit clear-linux-native-<kernel-version>.conf, and add iommu=pt intel_iommu=on to the end of the line.

  3. Then umount and reboot the machine.

    # cd /
# systemctl stop boot.mount
# reboot

  4. Set number of hugepages.

    # echo 1024 > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages

  5. Allocate pages on NUMA machines.

    # echo 1024 > /sys/devices/system/node/node0/hugepages/hugepages-2048kB/nr_hugepages
# echo 1024 > /sys/devices/system/node/node1/hugepages/hugepages-2048kB/nr_hugepages

  6. Making memory available for DPDK.

    # mkdir -p /mnt/huge
# mount -t hugetlbfs nodev /mnt/huge

  7. Add cores and memory configuration to the OpenvSwitch example; the /usr/share/openvswitch/scripts/ovs-ctl file can be edited on line 256 and DPDK configuration can be added.

    It should look something like: set ovs-vswitchd --dpdk -c 0x2 -n 4 --socket-mem 2048 -- unix:"$DB_SOCK"

    The next regular expression could be helpful:

    # sed -i s/"set ovs-vswitchd unix:"/"set ovs-vswitchd --dpdk -c 0x2 -n 4 --socket-mem 2048 -- unix:"/g /usr/share/openvswitch/scripts/ovs-ctl

  8. Start the OpenvSwitch service:

    # systemctl start openvswitch.service

  9. Create a virtual bridge using openvswitch:

    # ovs-vsctl add-br br0 -- set bridge br0 datapath_type=netdev

  10. Add dpdk ports to the bridge:

    # ovs-vsctl add-port br0 vhost-user1 -- set Interface vhost-user1 type=dpdkvhostuser
    # ovs-vsctl add-port br0 vhost-user2 -- set Interface vhost-user2 type=dpdkvhostuser

  11. Run guest virtual machine A using the next configuration as reference, where $IMAGE var is the name of the Clear Linux* OS for Intel Architecture image.

    qemu-system-x86_64 \
    -enable-kvm -m 1024 \
    -bios OVMF.fd \
    -smp 4 -cpu host \
    -vga none -nographic \
    -drive file="$IMAGE",if=virtio,aio=threads \
    -chardev socket,id=char1,path=/run/openvswitch/vhost-user1 \
    -netdev type=vhost-user,id=mynet1,chardev=char1,vhostforce \
    -device virtio-net-pci,mac=00:00:00:00:00:01,netdev=mynet1 \
    -object memory-backend-file,id=mem,size=1024M,mem-path=/dev/ hugepages,share=on \
    -numa node,memdev=mem -mem-prealloc \
    -debugcon file:debug.log -global isa-debugcon.iobase=0x402

  12. Run guest virtual machine B, use the configuration from the previous step; simply change the MAC address and the port socket. You can use 00:00:00:00:00:02 as a MAC address and vhost-user2 as a socket.

  13. Follow the instructions from the Setting IP address section.

Setting IP address

  1. Set an IP address to virtual machine for virtual machine A:

    # ip addr add dev enp0s2 10.0.0.5/24

    for virtual machine B:

    # ip addr add dev enp0s2 10.0.0.6/24

  2. Check if there is communication between both virtual machines using ping tool.

  3. Verify that Apache service is running:

    # systemctl status httpd.service
# systemctl start httpd.service

    Start httpd service only if it is inactive. Use apache benchmark to get information about the network performance between both virtual machines.

    # ab -n 1000000 -c 100 http://10.0.0.6/
